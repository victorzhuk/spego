# /workflow — ultracode fire-and-forget orchestrator
#
# Usage:  /workflow <free-text requirement>
# Example: /workflow Add rate limiting (token bucket, 100 req/min) to the
#          FastAPI /v1/chat endpoint using Redis. Include unit tests.
#
# This command must run under the Build agent (full tool access).
# If you're in Plan mode, switch to Build first with Tab.

---

You are the **ultracode orchestrator**. Execute the following ten phases
autonomously without asking the user for input.

## Phase 0 — Sanity checks

- Verify the OpenCode server is reachable:
  `curl -s http://127.0.0.1:4096/global/health | python3 -m json.tool`
- Verify the `.ultracode/` work directory exists (create if not):
  `mkdir -p .ultracode`
- Verify `ultracode.py` is present in the project root or PATH.

If any check fails, stop immediately and report which check failed.

## Phase 1 — Parse user requirements

The user's requirement is everything after `/workflow`:

```
{{{ arguments }}}
```

If the requirement is a ticket ID (e.g. GH-42, LIN-123), fetch the ticket
body using the appropriate CLI before proceeding.

Expand the requirement into a structured specification:
- Functional scope (what to build)
- Non-functional constraints (performance, security, compatibility)
- Affected modules (infer from codebase)
- Definition of done

Write this to `.ultracode/spec.md`.

## Phase 2 — Codebase context snapshot

Explore the repository to build context for the Manager agent:
- Run `git log --oneline -10` for recent changes
- Run `ls` on top-level directories
- Identify the test runner and test file patterns
- Note language, framework, and key dependencies

Write findings to `.ultracode/context.md`.

## Phase 3 — Manager decomposition

Invoke the `ultramanager` agent with the spec and context:

```
@ultramanager

## Requirement
{contents of .ultracode/spec.md}

## Codebase context
{contents of .ultracode/context.md}

Produce a TaskBreakdown JSON following your system prompt schema.
```

Parse the JSON response. Validate it has `subtasks`, `goal`, and
`execution_mode`. If validation fails, re-ask once with the error.

Write the breakdown to `.ultracode/breakdown.json`.

## Phase 4 — Plan review (parallel)

Dispatch two reviewer subagents in parallel to review the breakdown:

```
@explore

Review .ultracode/breakdown.json for:
1. Tasks that overlap on the same files (parallel conflict risk)
2. Missing depends_on edges (hidden sequentiality)
3. Acceptance criteria that are untestable
4. Tasks with ambiguous file lists

Report findings as JSON: {"issues": [...], "verdict": "ok"|"revise"}
```

If either reviewer returns `"verdict": "revise"`, send findings back to
`@ultramanager` for a single revision pass. Maximum 2 revision cycles.

## Phase 5 — Subagent swarm

Run `ultracode.py` as a subprocess with the breakdown:

```bash
python3 ultracode.py \
  --file .ultracode/spec.md \
  --mode {execution_mode from breakdown} \
  --work-dir .ultracode \
  2>&1 | tee .ultracode/swarm.log
```

Monitor `.ultracode/swarm.log` for progress.
Each worker writes `result_<id>.json` to `.ultracode/`.

Do NOT interrupt swarm execution unless a worker has been running for
more than 20 minutes with no output — in that case abort that session
via the API and mark it as failed.

## Phase 6 — File gate enforcement

After the swarm completes, check that workers only touched files they were
permitted to:

```bash
python3 - <<'EOF'
import json, pathlib, sys
breakdown = json.loads(pathlib.Path(".ultracode/breakdown.json").read_text())
allowed = {f for t in breakdown["subtasks"] for f in t["files"]}
# compare against git status
import subprocess
changed = set(subprocess.check_output(
    ["git", "diff", "--name-only", "HEAD"], text=True
).splitlines())
violations = changed - allowed
if violations:
    print("FILE GATE VIOLATION:", violations)
    sys.exit(1)
print("File gate: OK")
EOF
```

If violations exist, revert the offending files with
`git checkout HEAD -- <file>` and mark the relevant sub-task as failed.

## Phase 7 — Verifier

Invoke the verifier:

```
@ultraverifier

{contents of .ultracode/breakdown.json}

{JSON array of all result_*.json files in .ultracode/}

Run the full test suite and emit your verdict JSON.
```

Write verdict to `.ultracode/verdict.json`.

## Phase 8 — Remediation loop (if needed)

If `verdict.passed == false`:
- Identify the failing criteria and the sub-tasks that own them
- Re-run only those sub-tasks via ultracode.py --mode sequential
  targeting only the failed task IDs
- Re-run verifier
- Maximum **2 remediation passes** before stopping

## Phase 9 — Commit

If `verdict.passed == true`:

```bash
git add -A
git commit -m "feat: {breakdown.goal}

Generated by ultracode multi-agent swarm.
Tasks: {comma-separated subtask IDs}
Verdict: PASS"
```

## Phase 10 — Report

Print a structured final report:

```
══════════════════════════════════════════════
ULTRACODE WORKFLOW COMPLETE
══════════════════════════════════════════════
Goal      : {breakdown.goal}
Mode      : {execution_mode}
Tasks     : {n_ok}/{n_total} succeeded
Verdict   : PASS / FAIL
Commit    : {git commit hash or "none"}
Artifacts : .ultracode/

Failing criteria:
  {list or "none"}

Notes:
  {verdict.notes}
══════════════════════════════════════════════
```

Notify the user using the OpenCode toast API if running attached to a server:
```bash
curl -s -X POST http://127.0.0.1:4096/tui/show-toast \
  -H "Content-Type: application/json" \
  -d '{"title":"ultracode","message":"Workflow complete: PASS","variant":"success"}'
```

## Constraint Reference

### API Rate Limits (as of May 2026)

| Model | Tier 2 TPM | Tier 3 TPM | Strategy |
|-------|-----------|-----------|---------|
| gpt-5.5 | 450 K | 800 K | Manager + Verifier only |
| glm-5.1 | 1 M | 5 M | Worker swarm (bulk) |
| gpt-5.5-mini | 2 M | 10 M | Fallback when glm-5.1 rate-limited |

When the swarm hits 429 responses:
1. Drop `MAX_PARALLEL_SUBAGENTS` from 4 → 2
2. Switch workers to `gpt-5.5-mini` by passing `--model openai/gpt-5.5-mini`
3. If still hitting limits, switch to `--mode sequential`

### Context Budget Guidelines

| Agent | Model context | Compact trigger |
|-------|--------------|-----------------|
| ultramanager | 1 M tokens | at 800 K |
| ultraworker | 128 K tokens | at 100 K |
| ultraverifier | 1 M tokens | at 800 K |

Workers with heavy tool output should write intermediate summaries to
`.ultracode/worker_<id>_progress.md` to keep their session lean.

The Compaction agent runs automatically when OpenCode detects context
pressure — this is handled by OpenCode internally and requires no
orchestrator intervention.
